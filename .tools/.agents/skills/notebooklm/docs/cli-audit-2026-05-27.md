## Executive summary

There are no P0 issues in the CLI audit. The main gap is contract consistency, not basic functionality: the CLI has strong helper contracts for typed JSON error envelopes, Rich output routing, error handling, auth storage, and symlink safety, but those contracts are applied unevenly.

The **highest-risk user-facing issue** is that the `handle_errors` decorator re-raises `click.ClickException` subclasses unconditionally in its `except click.ClickException: raise` clause, so post-parse `UsageError`/`BadParameter` raised from inside command and service bodies bypass the typed JSON error envelope even when `--json` is active. This affects **five commands plus one helper** (P1#2) across both the command layer and the service layer - the broader contract gap of which `auth inspect`'s browser-prose path (P1#1) is one very visible instance.

The **highest-risk maintainability issue** is that the ADR-008 service-extraction boundary is only partially enforced, so new service modules can accumulate Click, rendering, runtime, or `exit_with_code` dependencies without tripping a guard. P1#1, P1#2, and the ADR-008 boundary finding converge on the same architectural seam: the contract decision driving P1#2's fix shape is the same decision that drives ADR-008 boundary expansion.

## Findings

### P1 - command-body and service-body `ClickException` bypasses the typed JSON error envelope

`handle_errors(...)` deliberately re-raises `click.ClickException` in its `except click.ClickException: raise` clause. That is fine for parse-time errors (Click's argument parser surfaces these before `--json` has been parsed). But many `UsageError`/`BadParameter` branches are raised *after* `--json` has been parsed, from inside command or service bodies, and those still emit Click usage text rather than a typed JSON error envelope.

**Command-layer sites** (five commands plus one helper):

- download flag conflicts: `src/notebooklm/cli/services/download.py:257`
- generate video style conflicts: `src/notebooklm/cli/services/generate.py:394`
- research wait cited-only conflict: `src/notebooklm/cli/research_cmd.py:158`
- source research cited-only conflict: `src/notebooklm/cli/source_cmd.py:626`
- chat conversation conflict: `src/notebooklm/cli/chat_cmd.py:193`
- language validation helper (called by `generate audio` and `generate video`): `src/notebooklm/cli/generate_cmd.py:57`

**Service-layer sites** (same shape, same contract gap):

- `src/notebooklm/cli/services/generate.py:383, 394, 396, 398` - `click.UsageError` raised from service code
- `src/notebooklm/cli/services/download.py:257, 259, 261` - same
- `src/notebooklm/cli/services/source_mutations.py:18` - imports `click` directly to support a `click.confirm` callable

This service-layer surface is precisely the architectural boundary that **[ADR-008 (CLI services extraction pattern)](adr/0008-cli-services-extraction-pattern.md)** polices (see the P2 ADR-008 finding below). The contract decision for the typed JSON error envelope and the ADR-008 boundary expansion are one initiative spanning ~10 service modules, not two adjacent findings.

Observed with mocked auth/client:

```text
cli generate video ... --style custom --json
exit 2
stdout: empty
stderr: Usage: cli generate video ...
        Error: --style custom requires --style-prompt

cli research wait ... --cited-only --json
exit 2
stdout: empty
stderr: Usage: cli research wait ...
        Error: --cited-only requires --import-all

cli ask ... --new -c conv123 ... --json
exit 2
stdout: empty
stderr: Usage: cli ask ...
        Error: --new and --conversation-id are mutually exclusive...
```

The published exit-code documentation does not address this gap. `docs/cli-exit-codes.md:52` lists `click.UsageError`/`BadParameter` as exit-2 (non-JSON), and `docs/cli-exit-codes.md:64` opens the JSON-mode section describing typed envelopes for *library* exceptions enumerated on `lines 41-51` (`RateLimitError`, `AuthError`, etc.). The doc is **silent on post-parse `UsageError`/`BadParameter`** rather than internally contradictory - the question P1#2 is asking is one the current doc does not answer. **ADR-015** is intended to fill that silence by deciding the contract explicitly.

Impact: callers cannot treat `--json` as a reliable typed JSON error envelope for validation failures.

Recommendation: decide the contract explicitly (ADR-015). If `--json` means "all command-body and service-body failures produce a typed JSON error envelope", convert these validation sites to a CLI-domain validation error or call `output_error(...)` when `current_json_output()` is true. If Click usage text is intentionally excluded, update help/docs/tests so automation knows `--json` only covers runtime/application errors. Land this decision *before* shaping any patch (P1#1's fix shape depends on it).

### P1 - `auth inspect --json` can emit non-JSON error output

`auth inspect` accepts `--json`, but it calls `_enumerate_browser_accounts(...)` before `render_auth_inspect(...)` has a chance to produce a structured response (`src/notebooklm/cli/session_cmd.py:576`, `src/notebooklm/cli/session_cmd.py:598`). The browser-cookie helpers print Rich/prose to the stdout console and call `exit_with_code(1)` directly on several failure paths:

- unknown browser: `src/notebooklm/cli/services/login/browser_accounts.py:197`
- unsupported browser function: `src/notebooklm/cli/services/login/browser_accounts.py:217`
- cookie validation failure: `src/notebooklm/cli/services/login/cookie_jar.py:111`
- stale-cookie failure: `src/notebooklm/cli/services/login/cookie_jar.py:126`
- network failure: `src/notebooklm/cli/services/login/cookie_jar.py:143`

Reproduction:

```text
notebooklm auth inspect --browser not-a-browser --json
exit 1
stdout: Unknown browser: 'not-a-browser'
        Supported: arc, brave, chrome, ...
stderr: empty
```

Impact: automation that relies on `json.loads(stdout)` breaks on an advertised JSON-capable command.

This is one (very visible) instance of the broader contract gap described in the previous finding. The cookie/account helpers emit Rich prose to stdout and call `exit_with_code(1)` directly from below the command layer - the same fatal-output shape from a different code path, interacting with the same JSON-mode contract decision that drives P1#2.

Recommendation: make the cookie/account enumeration layer return typed outcomes or raise typed CLI-domain exceptions; let the command layer choose JSON vs Rich rendering. A narrower patch (passing `json_output` into these helpers and using `output_error(..., json_output=True)` for all fatal branches) is available, but the choice between the broader and narrower patch is contingent on the typed-JSON-error-envelope contract decision in P1#2 and ADR-008's boundary policy.

### P2 - central JSON purity sweeps cover 16 of 71 JSON-capable commands

Dynamic Click inventory found 71 command paths exposing `--json`. The central success sweep has 16 cases (`tests/unit/test_json_stdout_purity.py:297`) and the central error sweep has 16 cases (`tests/unit/test_json_error_exit.py:286`). The success test comment says new JSON commands should fail until added (`tests/unit/test_json_stdout_purity.py:238`), but the list is curated and not cross-checked against the actual Click tree.

The two lists are overlapping; neither subsumes the other. Concretely they are also disjoint at the command level - no command appears in both - so the union covers at most ~30 of 71 `--json` commands and no command has both success-path AND error-path coverage under the central sweeps.

Impact: a new or missed command can support `--json` without ever entering the broad contract tests; a command in the success sweep can ship a regression in its error path and not trip the central tests.

Recommendation: add a small inventory assertion - discover all commands with `--json`, compare them to an explicit allowlist of covered/waived paths, and fail when a command is unclassified. Keep the hand-built argv cases, but make coverage drift visible. Extend with a per-command success+error coverage check.

### P2 - ADR-008 service boundary is only partially enforced

ADR-008 says service modules should make CLI business logic unit-testable without driving Click, keep Click/rendering policy at the command edge, and use plain service functions/plans (`docs/adr/0008-cli-services-extraction-pattern.md:66`, `docs/adr/0008-cli-services-extraction-pattern.md:67`, `docs/adr/0008-cli-services-extraction-pattern.md:68`). The boundary guard explicitly limits enforcement to `GUARDED_PATHS` (`tests/unit/cli/test_services_boundary.py:9`), and the guarded set currently contains only:

- `cli/services/listing.py`
- `cli/services/source_content.py`

Transitional guards cover only `artifact_generation.py` and `source_mutations.py` (`tests/unit/cli/test_services_boundary.py:56`).

Several unguarded service modules still import Click, rendering, runtime, or error handling directly, including:

- `src/notebooklm/cli/services/download.py:31`
- `src/notebooklm/cli/services/source_listing.py:16`
- `src/notebooklm/cli/services/polling.py:12`
- `src/notebooklm/cli/services/source_research.py:15`
- `src/notebooklm/cli/services/session_context.py:35`
- `src/notebooklm/cli/services/generate.py:41`
- `src/notebooklm/cli/services/auth_diagnostics.py:28`
- multiple `src/notebooklm/cli/services/login/*` modules

Impact: the codebase has a stated architectural target but weak regression pressure; future service modules can grow presentation/exit dependencies without tripping a guard. The P1#2 contract decision and the ADR-008 boundary expansion are one initiative - the typed JSON error envelope cannot be enforced uniformly without the boundary covering all service modules that currently exit fatally.

Recommendation: expand the boundary guard into a full inventory with three states: clean, transitional-with-exact-violations, and explicitly waived. Add a sibling assertion in the same guard module: no service module both imports `rendering.console` and calls `exit_with_code(1)`. One test surface, two assertions. This keeps the current migration honest without demanding a single giant refactor.

### P2 - command and service modules that intersect the typed JSON error envelope or the ADR-008 boundary

Prioritise extraction by risk, not line count. The command and service modules below combine Click entry, workflow decisions, rendering, file I/O, and client construction; the ones that *also* intersect the typed JSON error envelope (P1#2) or the ADR-008 boundary (previous finding) should move first.

**Highest risk - already implicated by P1 findings or by the ADR-008 boundary:**

- `src/notebooklm/cli/session_cmd.py` (hosts `auth inspect`, the P1#1 surface)
- `src/notebooklm/cli/services/download.py` (P1#2 service-layer site)
- `src/notebooklm/cli/services/generate.py` (P1#2 service-layer site)
- `src/notebooklm/cli/generate_cmd.py` (P1#2 language-validation helper)
- `src/notebooklm/cli/research_cmd.py` (P1#2 site `research wait`)
- `src/notebooklm/cli/source_cmd.py` (P1#2 site `source research`; also writes output files inline at `src/notebooklm/cli/source_cmd.py:150`, `src/notebooklm/cli/source_cmd.py:161`, `src/notebooklm/cli/source_cmd.py:176` instead of routing through a service)
- `src/notebooklm/cli/chat_cmd.py` (P1#2 site)
- `src/notebooklm/cli/services/playwright_login.py` (auth-flow surface that also runs under `auth inspect`)

Line counts as secondary context (snapshot at HEAD `615d5b8b`):

- `src/notebooklm/cli/source_cmd.py`: 834 lines
- `src/notebooklm/cli/session_cmd.py`: 739 lines
- `src/notebooklm/cli/generate_cmd.py`: 686 lines
- `src/notebooklm/cli/artifact_cmd.py`: 598 lines

Impact: larger command modules make typed-JSON-error-envelope / Rich / quiet consistency harder to audit because output policy is spread across many direct `console.print(...)` branches and service helpers. The risk is concentrated where the modules cross the JSON contract or the ADR-008 boundary, not where the line counts are simply large.

Recommendation: move the validation/output paths in `auth inspect`, `download`, `generate video`, and `research wait` first, since they already intersect the JSON contract. Treat `playwright_login.py` as the next surface because it is the largest service module crossing the auth-flow boundary. Defer pure size-reduction work on the other modules until the JSON-contract risk has been retired.

### P2 - Rich/console policy exists, but bypasses remain common

`rendering.emit_status(...)` has the right contract: in JSON mode status goes to stderr, and root `--quiet` suppresses status (`src/notebooklm/cli/rendering.py:38`, `src/notebooklm/cli/rendering.py:57`, `src/notebooklm/cli/rendering.py:69`). Some commands use it correctly, for example chat save-as-note status (`src/notebooklm/cli/chat_cmd.py:296`, `src/notebooklm/cli/chat_cmd.py:326`).

But many CLI and service paths still call `console.print(...)` directly. Some are final human output and are fine; others are diagnostics/fatal branches that should be policy-routed. The login cookie helpers are the clearest offender because they can run under `auth inspect --json`.

Impact: reviewers must reason path-by-path about stdout, stderr, JSON mode, and quiet mode instead of relying on helper usage.

Recommendation: document a short rule - command data output may use `console.print`; status/diagnostic/fatal output must use `emit_status`, `cli_print`, or `output_error`. The enforcement belongs in the same boundary guard described under the ADR-008 finding (one test surface, two assertions).

### P2 - project-scope `skill install` overwrites agent skill files without confirmation

`skill install --scope project` resolves the scope root to `Path.cwd()` (`src/notebooklm/cli/skill_cmd.py:56`) and writes packaged skill content directly to fixed agent paths (`src/notebooklm/cli/skill_cmd.py:144`). That is not arbitrary path traversal, but it is a workspace mutation into agent-readable configuration without `--dry-run`, `--yes`, backup, or no-clobber behavior.

Impact: accidental overwrite of local agent skill customizations; in a compromised package scenario, agent-facing instructions are written into the project.

Recommendation: add `--dry-run`, detect existing differing content, and require `--yes` or `--force` for overwrites in project scope. At minimum, print the target paths before writing and support `--no-clobber`. Note that one invocation writes BOTH `.claude/skills/notebooklm/SKILL.md` AND `.agents/skills/notebooklm/SKILL.md` simultaneously - add per-target `--no-clobber` semantics rather than a single global flag.

### Strengths observed

- Source uploads reject symlinks by default to reduce accidental exfiltration (`src/notebooklm/cli/services/source_add.py:82`, `src/notebooklm/cli/source_cmd.py:332`).
- Playwright login filters persisted cookies by domain policy (`src/notebooklm/cli/services/playwright_login.py:142`, `src/notebooklm/cli/services/playwright_login.py:801`).
- Secret-bearing JSON writes go through `atomic_write_json(..., mode=0o600)` in `src/notebooklm/_atomic_io.py`.
- Login path directories are created with restrictive modes on POSIX (`src/notebooklm/cli/services/playwright_login.py:540`).
- Unexpected error rendering avoids stringifying arbitrary full exception state and logs details only to the redacting debug logger in the generic `except Exception` branch. **Cross-reference:** this strength is in the *same module* as the P1#2 root cause - the `ClickException` re-raise branch is what allows post-parse `UsageError`/`BadParameter` to bypass the typed JSON error envelope. The strength applies to the unexpected-exception path, not to the `ClickException` re-raise path.
- Unit CLI suite is broad and currently green: 1527 tests passed.
